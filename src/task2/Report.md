
## 1. Общие результаты

### Сценарий 1 — «Утренний час пик»: 50 000 операций find по случайным кодам из датасета

```
AVL ~ 0.027 sec
List ~ 0.653 sec
```

### Сценарий 2 — «Обновление базы»: 10 000 операций insert с новыми кодами, перемежающихся с 10 000 операций find

```
AVL ~ 0.022
List ~ 0.133
```

## 2. `perf report` для `List`

### Сценарий 1

    Наиболее неэффективной у List оказалась функция find (~ 23%). Произошло это из-за сложности поиска у списка - O(n), в то время как у AVL - O(log n)

### Сценарий 2

    Наиболее неэффективной также у List оказалась также find (~ 30%). Несмотря на то, что в это сценарии активно использовалась функция insert, она не попала в этот рейтинг, так по специфике реализации вставка происходила в начало списка - O(1), у AVL - O(log n). Несмотря даже на такую быструю вставку, List проигрывает по скорости вставке и поиска AVL вцелом.

## 3. Причины разницы

    AVL поддерживается сбалансированной структурой, поэтому вставка и поиск происходит за O(log n). List реализован так, что вставка происходит за O(1), но поиск линейный, то есть за O(n). Поэтому в сценарии 1 по производительности выигрывает AVL. Сценарий 2 учитывает вставку и поиск в совокупности, поэтому List тут не получает преимущество со своей скоростью вставки.

## 4. Итоговый отчет

    По результатам сравнительного тестирования архитектура на базе АВЛ-дерева продемонстрировала многократное превосходство в скорости обработки запросов по сравнению с классическим списком. Данное решение обеспечивает мгновенный поиск по базе аэропортов вне зависимости от её объема, что гарантирует стабильную работу системы регистрации даже в периоды пиковых нагрузок.
